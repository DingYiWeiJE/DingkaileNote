## 表情占用多个索引

```ts
function getCodePointIndexMap(inputString: string) {
    // 使用 Map 来存储 Code Point 索引 (key) 到 Code Unit 索引 (value) 的映射
    let indexMap = new Map();
    
    // Code Point 索引：表示当前是字符串中的第几个字符（基于 Unicode 字符计数）
    let codePointIndex = 0;
    
    // Code Unit 索引：表示当前字符在底层存储（UTF-16）中的起始位置
    let codeUnitIndex = 0;
    
    // 使用 for...of 循环迭代字符串。
    // 在 JavaScript 中，for...of 会按完整的 Unicode 字符（Code Point）进行迭代。
    for (const char of inputString) {
        // 1. 记录映射：
        // 记录当前 Code Point 索引 (0, 1, 2...) 对应的 Code Unit 索引 (0, 1, 2...)
        indexMap.set(codePointIndex, codeUnitIndex);
        
        // 2. 更新 Code Point 索引：
        // 每次循环迭代一个完整的 Code Point，所以总是 +1
        codePointIndex += 1;
        
        // 3. 更新 Code Unit 索引：
        // char.length 返回该字符在 UTF-16 中占用的 Code Unit 数量。
        // - 对于 BMP 字符 ('a', '你'): char.length 为 1
        // - 对于非 BMP 字符 ('😀', '🌍'): char.length 为 2 (代理对)
        codeUnitIndex += char.length;
    }
    
    return indexMap;
}
```

## 请求头

`Content-Type` 是 HTTP 头部的一个字段，用于指示发送的内容的类型。常见的 `Content-Type` 类型有以下几种：

1. **text/plain**
    纯文本文件。它表示内容为没有格式的纯文本。
2. **text/html**
    HTML 文档。表示返回的数据是一个 HTML 页面。
3. **application/json**
    JSON 格式数据。用于表示 JSON 格式的内容，通常用于 API 响应。
4. **application/xml**
    XML 格式数据。用于表示 XML 格式的内容，常用于某些 Web 服务或 API。
5. **multipart/form-data**
    用于表单上传文件时，表示提交的数据包含文件字段。通常与文件上传相关。
6. **application/x-www-form-urlencoded**
    表单数据编码类型，通常用于简单的表单提交，数据以 `key=value` 的形式发送，且值通过 URL 编码。
7. **application/javascript**
    JavaScript 脚本文件。表示内容是 JavaScript 代码。
8. **image/jpeg**
    JPEG 格式的图片。表示返回的数据是 JPEG 格式的图片文件。
9. **image/png**
    PNG 格式的图片。表示返回的数据是 PNG 格式的图片文件。
10. **audio/mpeg**
     MPEG 音频格式，通常用于 MP3 音频文件。
11. **video/mp4**
     MP4 视频格式，通常用于视频文件。
12. **application/pdf**
     PDF 文件。用于指示返回的数据是 PDF 格式的文件。
13. **application/octet-stream**
     二进制流数据。常用于表示不特定类型的二进制数据，或用于文件下载。
14. **application/zip**
     ZIP 压缩文件。表示返回的是一个 ZIP 文件。

这些都是常见的 `Content-Type` 类型，实际上根据不同的需求和协议，还可能会有其他更具体的类型。

## Promise链

```js
let chain = Promise.resolve()
  for (let i = 0; i < 3; i++) {
    chain = chain.then(() => {
      console.log(`inside chain ${i}`)
    })
  }
```

由于 `chain` 被不断更新，新的 `.then` 回调会等待前一个 `.then` 回调执行完毕后再执行。因此，输出会是顺序的：

```js
let chain = Promise.resolve()
  for (let i = 0; i < 3; i++) {
    chain.then(() => {
      console.log(`inside chain ${i}`)
    })
  }
```

由于所有的 `.then` 回调都是直接附加到初始的 `Promise.resolve()` 上，它们会几乎同时执行（因为初始 Promise 已经 resolved），而不是按顺序执行。

## 代理服务器获取图片

```js
import express from 'express';
import fetch from 'node-fetch';
import cors from 'cors';  // 引入 CORS 中间件

const app = express();
const port = 9527;

// 启用 CORS，允许所有来源22
app.use(cors());

app.get('/proxy', async (req, res) => {
  // 获取请求参数中的 imgUrl
  const imgUrl = req.query.imgUrl;

  if (!imgUrl) {
    return res.status(400).send('Missing imgUrl parameter');
  }

  try {
    // 从传入的 imgUrl 请求图片
    const response = await fetch(imgUrl);

    // 检查响应是否为图片类型
    const contentType = response.headers.get('content-type');
    if (!contentType || !contentType.startsWith('image')) {
      return res.status(400).send('Provided URL is not an image');
    }

    // 使用 arrayBuffer 获取图片数据并返回
    const arrayBuffer = await response.arrayBuffer();
    const buffer = Buffer.from(arrayBuffer);  // 将 ArrayBuffer 转换为 Node.js Buffer

    res.set('Content-Type', contentType);  // 设置正确的 Content-Type
    res.send(buffer);
  } catch (err) {
    res.status(500).send('Failed to fetch image');
  }
});

app.listen(port, () => {
  console.log(`Server running at http://localhost:${port}`);
});
```

## 限制并发请求数量

```js
export function concurPromise<T>(promises: Promise<T | Error>[], maxNum: number): Promise<(T | Error)[]> {
  return new Promise((resolve) => {
    if (promises.length === 0) {
      return resolve([]);
    }

    const results: (T | Error)[] = new Array(promises.length);
    let index = 0;
    let activeCount = 0;

    const runPromise = async (i: number) => {
      activeCount++;
      try {
        const res = await promises[i];
        results[i] = res;
      } catch (error) {
        results[i] = error instanceof Error ? error : new Error(String(error));
      } finally {
        activeCount--;
        if (activeCount === 0 && index >= promises.length) {
          resolve(results);
        } else {
          dequeue();
        }
      }
    };

    const dequeue = () => {
      while (activeCount < maxNum && index < promises.length) {
        runPromise(index++);
      }
    };
    dequeue();
  });
}
```

## 正则表达式

### 概念

`.`表示任意字符

`\`表示转义字符

`?`表示0个或者1个

`()`表示分组，在使用match的时候，会把分组内容输出出来的

```js
const str = ".."
console.log(str.match(/(\.?)\.$/))
// [ '..', '.', index: 0, input: '..', groups: undefined ]
// 会先输出匹配到的数据'..'， 然后返回匹配到的分组的内容'.'
// 如果不想让其输出，则在（）中加上 ？：
console.log(str.match(/(?:\.?)\.$/))
// [ '..', index: 0, input: '..', groups: undefined ]
```

`^`表示一个空格

```js
const str = "/.."
console.log(str.match(/(?:^|\/)\.?\.$/))
// (?:^|\/) 这一段做了或的处理，也就是说，匹配到一个空格，或者一个 \ 都可以
// 所以str = ".." 也能匹配上
```

### 配置字体文件

#### web

1. **`truetype` (TTF)**:
   - 标准的字体格式，支持平滑缩放，常用于操作系统和网络上。大多数浏览器都能支持 TTF 格式，但它比其他格式（如 WOFF）文件要大，因此加载时可能稍慢。
2. **`woff` (Web Open Font Format)**:
   - 专为 Web 设计的字体格式，支持更好的压缩和优化，能够提高加载速度。现代浏览器都支持这个格式。
3. **`woff2`**:
   - 这是 WOFF 的更新版本，比 WOFF 具有更好的压缩率，能有效减少文件大小，因此加载速度更快。
4. **`eot` (Embedded OpenType)**:
   - 主要用于 Internet Explorer（IE）浏览器。现代浏览器很少需要它，因为它已被更现代的格式（如 WOFF）所取代。
5. **`svg`**:
   - 主要用于移动设备和早期版本的浏览器，尤其是 iOS 的 Safari 浏览器。

```css
@font-face {
  font-family: 'MyFont';
  src: url('./my-font.ttf') format('truetype'),
    url('./my-font2.ttf') format('truetype');
  font-weight: 600;
  font-display: swap;
  font-style: normal;
  
}
```

`font-display` 是一个 CSS 属性，用于控制字体加载时的显示方式。它不会直接做到“懒加载”，但可以有效减少因字体加载慢导致的“无字体”问题，从而优化用户体验。

- **`font-display: swap`**：当字体还没加载时，浏览器会使用备用字体显示文本，等字体加载完成后再替换为目标字体。这能避免字体加载时的空白或无字体显示（FOIT）。
- **`font-display: optional`**：该值会尽量不等待字体加载。如果字体加载不及时，浏览器会使用备用字体，并在字体加载完成后替换。对性能优化有帮助，特别是在字体资源较大的情况下。

## 观察dom是否可见的钩子

```ts
import { useState, useEffect, useCallback } from 'react';

function useInViewport(
  ref: React.RefObject<HTMLElement>,
  options: IntersectionObserverInit = { threshold: 0.1 }
): boolean {
  const [isInViewport, setIsInViewport] = useState(false);

  const handleObserverUpdate = useCallback(
    ([entry]: IntersectionObserverEntry[]) => {
      setIsInViewport(entry.isIntersecting);
    },
    []
  );

  useEffect(() => {
    const node = ref.current;
    if (!node) return;

    const observer = new IntersectionObserver(handleObserverUpdate, options);
    observer.observe(node);

    return () => {
      observer.disconnect(); // 更安全地释放 observer
    };
  }, [ref, handleObserverUpdate, options]);

  return isInViewport;
}

export default useInViewport;

```

```ts
    listeners.push(listener)

    return function unsubscribe() {
      const index = listeners.indexOf(listener)
      if (index !== -1) {
        listeners.splice(index, 1)
      }
    }
  }

  function getState(): T {
    return state
  }

  function setState(newState: T | UpdateFun<T>) {
    let nextState: T
    if (typeof newState === 'function') {
      nextState = (newState as UpdateFun<T>)(state)
    } else {
      nextState = newState 
    }
    state = nextState
    listeners.forEach(listener => listener(state))
  }

  return {subscribe, getState, setState}
}

```

### Crash: Screen Snapshot Crash

  原因分析：

  从stack trace的关键调用链：
  RNSScreen setViewToSnapshot
    ↓
  UIView resizableSnapshotViewFromRect
    ↓
  CARenderServerSnapshot (系统截图API)
    ↓
  mach_msg2_trap (系统底层IPC调用)

  这个crash发生在 react-native-screens 库尝试对屏幕进行截图时：

  1. freezeOnBlur 机制：当屏幕失去焦点时，为了优化性能，react-native-screens 会冻结该屏幕（通过截图保存状态）
  2. crash触发条件：

    - 屏幕正在快速切换/卸载时
    - 视图层级还未完全渲染完成
    - iOS系统的 CARenderServerSnapshot API 在这种时机被调用会失败
  3. 为什么崩溃：截图操作是同步的系统调用，如果此时视图不在可截图状态，会触发系统级异常

  修复方法：
  freezeOnBlur: false  // 禁用冻结优化


### Turbo Module Crash

在根目录的_layout.tsx中， 是直接和底层交互的，所以其中执行的所有原生模块调用都应该try-catch

  原因分析：

  Stack trace显示：
  ObjCTurboModule::performVoidMethodInvocation
    ↓
  objc_exception_rethrow (异常重新抛出)
    ↓
  std::__terminate (C++异常终止)
    ↓
  abort (程序中止)

  这是典型的未捕获原生模块异常：

  1. Turbo Module：React Native新架构中，JS和原生代码之间的桥接层
  2. crash触发：

    - JS调用原生模块方法（如通知、GoogleSignIn等）
    - 原生代码内部抛出异常（如权限问题、配置错误、API调用失败）
    - 异常未被捕获，向上传播到C++层
    - C++调用 std::__terminate 终止程序
  3. 可能原因：

    - 通知权限请求在某些设备/配置下失败
    - GoogleSignin配置在应用启动早期调用失败
    - 监听器注册/移除时对象已释放

### Hermes 指针报错

1. 堆栈追踪分析

  从崩溃堆栈可以看到关键信息：

  #0 hermes::vm::HiddenClass::getHasIndexLikeProperties()
  #1 hermes::vm::JSObject::defineOwnComputedPrimitive()

  这表明 Hermes 引擎在定义对象的计算属性时，检查对象是否具有"类似索引"的属性时崩溃了。

  堆栈还显示：
  #13 hermes::vm::arrayPrototypeForEach()

  说明崩溃发生在执行 forEach 循环时

**解决方案**

```
 1. Map 是专门的键值对集合
    - 不受 JavaScript 对象的"索引属性"限制
    - 任何值（包括数字字符串）都被视为普通的键
    - Hermes 不会对 Map 的键进行"类似索引"的检查
  2. 更安全的 API
    - map.get(key) 替代 obj[key] - 避免属性访问的歧义
    - map.set(key, value) 替代 obj[key] = value - 不触发对象内部结构重组
    - map.has(key) 替代 key in obj - 更明确的检查方式
  3. 性能更好
    - Map 针对频繁的增删操作优化
    - 不需要 Hermes 的 HiddenClass 优化机制
    - 避免了对象形状（shape）的变化
```

读取动态属性的时候， 不要用 object[key] 的方式， 而要用map的方式来解决

## watch

```ts
const name = ref('张三');
const age = ref(18);
const address = reactive({
  city: '北京',
  street: '长安街'
});

watch(name, (newVal, oldVal) => {
  console.log('name changed:', newVal, oldVal);
});

watch(address, (newVal, oldVal) => {
  console.log('address changed:', newVal, oldVal);
}, { deep: true, immediate: true });

watch(
  () => address.city,
  (newVal, oldVal) => {
    console.log('city changed:', newVal, oldVal);
  },
)
```

+ 值类型的数据，可以拿到oldvalue
+ 引用数据类型的数据，只有当整个值的地址发生了变化的时候才会拿到oldvalue
+ 监听引用数据类型时需要加上deep才能监听到内部属性的变化，否则只有当整个值的地址发生变化的时候才会触发；当这个值的层级很深的时候，会影响性能，可以传入deep：2 只侦听两层
+ immediate 属性值可以决定刚刚进页面的时候是否触发
+ 传入一个getter函数会比deep侦听性能更高，并且在没有更改引用地址的情况下也能获取到oldvalue
