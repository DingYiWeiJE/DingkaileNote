# 持久化储存

| **方案**                       | **对应 Web 概念**       | **特点**                                        | **适用场景**                       |
| ------------------------------ | ----------------------- | ----------------------------------------------- | ---------------------------------- |
| **DataStore (Preferences)**    | `localStorage`          | 基于协程 (Flow)，异步处理，支持简单键值对。     | 用户设置、登录状态、小型配置。     |
| **Room**                       | `IndexedDB` / `Web SQL` | 关系型数据库 (SQLite)，支持复杂查询和大量数据。 | 缓存列表数据、本地字典、离线文档。 |
| **EncryptedSharedPreferences** | 敏感型 `localStorage`   | 自动对存储内容进行硬件级加密。                  | 存储 Token、密钥等敏感信息。       |

### 创建 DAO（数据访问对象）

```kotlin
package com.huagong.controlx.data.local.db.communication

import androidx.room.Dao  // ← 添加这个 import
import androidx.room.Insert
import androidx.room.Query
import androidx.room.Update
import kotlinx.coroutines.flow.Flow

@Dao  // ← 添加这个注解
interface CommunicationLogDao {
    @Insert
    suspend fun insert(log: CommunicationLogEntity): Long

    @Update
    suspend fun update(log: CommunicationLogEntity)

    @Query("SELECT * FROM communication_logs ORDER BY createdAt DESC")
    fun getAllLogs(): Flow<List<CommunicationLogEntity>>

    @Query("SELECT * FROM communication_logs WHERE requestIsRuntime = 0 ORDER BY createdAt DESC")
    fun getFilteredLogs(): Flow<List<CommunicationLogEntity>>

    @Query("DELETE FROM communication_logs")
    suspend fun clearAll()

    @Query("SELECT * FROM communication_logs WHERE id = :id")
    suspend fun getById(id: Long): CommunicationLogEntity?
}
```

Room 需要通过 @Dao 注解来识别这是一个数据访问对象，然后在编译时自动生成实现代码。

  没有 @Dao：Room 不知道这是 DAO，不会生成代码 → 编译错误
  有 @Dao：Room 识别并生成实现代码 → 编译成功

#### Flow

```markdown
Flow vs SQL：概念对比                                                          
  传统 SQL 查询（一次性）                                   

  -- 执行一次，返回结果
  SELECT * FROM communication_logs ORDER BY createdAt DESC;

  特点：
  - 查询一次，得到一个结果
  - 数据变化了？需要再查一次
  - 就像拍照：只能看到那一瞬间的数据

  Flow 查询（持续监听）

  @Query("SELECT * FROM communication_logs ORDER BY createdAt DESC")
  fun getAllLogs(): Flow<List<CommunicationLogEntity>>

  特点：
  - 查询一次，持续接收结果
  - 数据变化了？自动推送新结果
  - 就像视频直播：实时看到数据变化

  ---
  Flow 的工作原理

  1. 建立连接（订阅）

  // ViewModel 中
  val logs: StateFlow<List<CommunicationLogEntity>> =
      dao.getAllLogs()  // 返回 Flow
          .stateIn(
              scope = viewModelScope,
              started = SharingStarted.WhileSubscribed(5000),
              initialValue = emptyList()
          )

  类比 SQL：
  -- 就像建立一个"实时视图"
  CREATE VIEW live_logs AS
  SELECT * FROM communication_logs ORDER BY createdAt DESC;

  -- 但这个视图会"自动刷新"！

  2. 接收数据（消费）

  // UI 中
  val logs by viewModel.logs.collectAsStateWithLifecycle()

  // 每当数据库变化，logs 自动更新
  LazyColumn {
      items(logs) { log ->
          LogCard(log)
      }
  }

  类比：
  数据库变化 → Flow 自动推送 → UI 自动更新

  就像：
  INSERT → 触发通知 → 界面刷新
  UPDATE → 触发通知 → 界面刷新
  DELETE → 触发通知 → 界面刷新

  ---
  Flow 的完整使用流程

  步骤 1：DAO 中定义（数据源）

  @Dao
  interface CommunicationLogDao {
      // 返回 Flow，表示"持续监听"
      @Query("SELECT * FROM communication_logs ORDER BY createdAt DESC")
      fun getAllLogs(): Flow<List<CommunicationLogEntity>>

      // 对比：不用 Flow，只查一次
      @Query("SELECT * FROM communication_logs ORDER BY createdAt DESC")
      suspend fun getAllLogsOnce(): List<CommunicationLogEntity>
  }

  SQL 思维：
  -- Flow 版本：像一个"活的查询"
  SELECT * FROM communication_logs
  ORDER BY createdAt DESC
  -- 并且：当表数据变化时，自动重新执行

  -- 非 Flow 版本：普通查询
  SELECT * FROM communication_logs
  ORDER BY createdAt DESC
  -- 执行一次就结束

  步骤 2：ViewModel 中转换（中间层）

  @HiltViewModel
  class CommunicationLogViewModel @Inject constructor(
      private val dao: CommunicationLogDao
  ) : ViewModel() {

      // 将 Flow 转换为 StateFlow（UI 友好的格式）
      val logs: StateFlow<List<CommunicationLogEntity>> =
          dao.getAllLogs()
              .stateIn(
                  scope = viewModelScope,
                  started = SharingStarted.WhileSubscribed(5000),
                  initialValue = emptyList()
              )
  }

  解释：
  - dao.getAllLogs()：从数据库获取 Flow
  - .stateIn(...)：转换为 StateFlow（带初始值，适合 UI）
  - viewModelScope：ViewModel 销毁时自动取消订阅

  步骤 3：UI 中消费（展示层）

  @Composable
  fun CommunicationLogScreen(
      viewModel: CommunicationLogViewModel = hiltViewModel()
  ) {
      // 收集 Flow 数据
      val logs by viewModel.logs.collectAsStateWithLifecycle()

      // 使用数据
      LazyColumn {
          items(logs) { log ->
              Text("${log.requestName} - ${log.requestTime}")
          }
      }
  }

  ---
  Flow 的操作符（类似 SQL 子句）

  1. 过滤（WHERE）

  // SQL
  SELECT * FROM communication_logs
  WHERE requestIsRuntime = 0;

  // Flow
  dao.getAllLogs()
      .map { logs ->
          logs.filter { !it.requestIsRuntime }
      }

  2. 转换（SELECT 特定字段）

  // SQL
  SELECT requestName, requestTime FROM communication_logs;

  // Flow
  dao.getAllLogs()
      .map { logs ->
          logs.map { log ->
              "${log.requestName} at ${log.requestTime}"
          }
      }

  3. 限制数量（LIMIT）

  // SQL
  SELECT * FROM communication_logs LIMIT 10;

  // Flow
  dao.getAllLogs()
      .map { logs -> logs.take(10) }

  ---
  实际例子：完整的数据流

  场景：插入一条日志，UI 自动更新

  // 1. DAO 定义
  @Dao
  interface CommunicationLogDao {
      @Query("SELECT * FROM communication_logs ORDER BY createdAt DESC")
      fun getAllLogs(): Flow<List<CommunicationLogEntity>>

      @Insert
      suspend fun insert(log: CommunicationLogEntity)
  }

  // 2. ViewModel
  @HiltViewModel
  class CommunicationLogViewModel @Inject constructor(
      private val dao: CommunicationLogDao
  ) : ViewModel() {

      val logs = dao.getAllLogs()
          .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyList())

      fun addLog(name: String, data: String) {
          viewModelScope.launch {
              dao.insert(
                  CommunicationLogEntity(
                      requestName = name,
                      requestIsRuntime = false,
                      requestTime = getCurrentTime(),
                      requestData = data
                  )
              )
              // 插入后，Flow 自动推送新数据，UI 自动更新！
          }
      }
  }

  // 3. UI
  @Composable
  fun LogScreen(viewModel: CommunicationLogViewModel = hiltViewModel()) {
      val logs by viewModel.logs.collectAsStateWithLifecycle()

      Column {
          Button(onClick = {
              viewModel.addLog("测试", """{"test": true}""")
          }) {
              Text("添加日志")
          }

          // 点击按钮后，这个列表会自动更新！
          LazyColumn {
              items(logs) { log ->
                  Text(log.requestName)
              }
          }
      }
  }

  数据流动：
  用户点击按钮
      ↓
  viewModel.addLog()
      ↓
  dao.insert() 插入数据库
      ↓
  数据库触发变化通知
      ↓
  Flow 自动推送新数据
      ↓
  StateFlow 更新
      ↓
  UI 自动重组（recompose）
      ↓
  列表显示新数据

  ---
  Flow vs 传统方式对比

  传统方式（手动刷新）

  // ❌ 传统方式
  class OldViewModel {
      private val _logs = MutableLiveData<List<Log>>()
      val logs: LiveData<List<Log>> = _logs

      fun loadLogs() {
          viewModelScope.launch {
              _logs.value = dao.getAllLogsOnce()  // 手动查询
          }
      }

      fun addLog(log: Log) {
          viewModelScope.launch {
              dao.insert(log)
              loadLogs()  // 手动刷新！
          }
      }
  }

  问题：
  - 每次修改数据后，必须手动调用 loadLogs()
  - 如果忘记调用，UI 不会更新
  - 多个地方修改数据，容易遗漏刷新

  Flow 方式（自动刷新）

  // ✅ Flow 方式
  class NewViewModel {
      val logs = dao.getAllLogs()  // 自动监听
          .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyList())

      fun addLog(log: Log) {
          viewModelScope.launch {
              dao.insert(log)
              // 不需要手动刷新，Flow 自动推送！
          }
      }
  }

  优势：
  - ✅ 插入、更新、删除后自动刷新
  - ✅ 不会忘记刷新
  - ✅ 代码更简洁

  ---
  常见问题

  Q1: Flow 会不会一直查询数据库，很耗性能？

  A: 不会！Room 很聪明：
  - 只有数据真正变化时才推送
  - 使用了数据库的"触发器"机制
  - 不是轮询，是事件驱动

  Q2: 如果只想查一次怎么办？

  A: 用 suspend 函数：
  @Query("SELECT * FROM communication_logs")
  suspend fun getAllLogsOnce(): List<CommunicationLogEntity>

  Q3: Flow 什么时候停止监听？

  A: 当 viewModelScope 销毁时（ViewModel 销毁时）自动停止

  ---
  总结：SQL vs Flow

  ┌──────────┬───────────────┬────────────────┐
  │ SQL 概念 │   Flow 对应   │      说明      │
  ├──────────┼───────────────┼────────────────┤
  │ SELECT   │ @Query        │ 查询数据       │
  ├──────────┼───────────────┼────────────────┤
  │ WHERE    │ .filter { }   │ 过滤数据       │
  ├──────────┼───────────────┼────────────────┤
  │ ORDER BY │ .sortedBy { } │ 排序           │
  ├──────────┼───────────────┼────────────────┤
  ├──────────┼───────────────┼────────────────┤
  │ 触发器   │ Flow 自动推送 │ 数据变化时通知 │
  ├──────────┼───────────────┼────────────────┤
  │ 视图     │ Flow          │ 实时查询结果   │
  └──────────┴───────────────┴────────────────┘

  ---
  现在理解了吗？

  Flow 就是：
  - ✅ 一个"活的 SQL 查询"
  - ✅ 数据变化时自动推送
  - ✅ 不需要手动刷新
```

### 创建viewModel

```kotlin
package com.huagong.controlx.presentation.profile

import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import com.huagong.controlx.data.local.db.communication.CommunicationLogDao
import com.huagong.controlx.data.local.db.communication.CommunicationLogEntity
import com.huagong.controlx.data.local.db.communication.LogStatus
import dagger.hilt.android.lifecycle.HiltViewModel
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.SharingStarted
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow
import kotlinx.coroutines.flow.flatMapLatest
import kotlinx.coroutines.flow.stateIn
import kotlinx.coroutines.launch
import java.text.SimpleDateFormat
import java.util.Date
import java.util.Locale
import javax.inject.Inject

@HiltViewModel
class CommunicationLogViewModel @Inject constructor(
    private val dao: CommunicationLogDao
) : ViewModel() {

    private val _filterRuntimeData = MutableStateFlow(false)
    val filterRuntimeData: StateFlow<Boolean> = _filterRuntimeData.asStateFlow()

    val logs: StateFlow<List<CommunicationLogEntity>> = _filterRuntimeData
    .flatMapLatest { filter ->
                    if (filter) {
                        dao.getFilteredLogs()
                    } else {
                        dao.getAllLogs()
                    }
                   }
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5000),
        initialValue = emptyList()
    )

    fun toggleFilter() {
        _filterRuntimeData.value = !_filterRuntimeData.value
    }

    fun logRequest(
        name: String,
        data: String,
        isRuntime: Boolean = false
    ): Long {
        var logId: Long = 0
        viewModelScope.launch {
            logId = dao.insert(
                CommunicationLogEntity(
                    requestName = name,
                    requestIsRuntime = isRuntime,
                    requestTime = getCurrentTime(),
                    requestData = data,
                    status = LogStatus.PENDING
                )
            )
        }
        return logId
    }

    fun logResponse(
        logId: Long,
        name: String,
        data: String,
        isRuntime: Boolean = false
    ) {
        viewModelScope.launch {
            val log = dao.getById(logId)
            log?.let {
                dao.update(
                    it.copy(
                        responseName = name,
                        responseIsRuntime = isRuntime,
                        responseTime = getCurrentTime(),
                        responseData = data,
                        status = LogStatus.COMPLETED
                    )
                )
            }
        }
    }

    fun clearAllLogs() {
        viewModelScope.launch {
            dao.clearAll()
        }
    }

    private fun getCurrentTime(): String {
        val sdf = SimpleDateFormat("HH:mm:ss", Locale.getDefault())
        return sdf.format(Date())
    }
}
```

#### 说明

```
代码详细解释

  1. @HiltViewModel 和构造函数注入

  @HiltViewModel
  class CommunicationLogViewModel @Inject constructor(
      private val dao: CommunicationLogDao
  ) : ViewModel()

  解释：
  - @HiltViewModel：告诉 Hilt 这是一个 ViewModel
  - @Inject constructor：Hilt 自动注入 dao
  - private val dao：保存 DAO 引用，用于数据库操作

  Hilt 的工作：
  1. UI 需要 CommunicationLogViewModel
     ↓
  2. Hilt 看到 @HiltViewModel
     ↓
  3. Hilt 调用 provideCommunicationLogDao() 获取 DAO
     ↓
  4. Hilt 创建 ViewModel，传入 DAO
     ↓
  5. 返回 ViewModel 给 UI

  ---
  2. 过滤状态管理

  private val _filterRuntimeData = MutableStateFlow(false)
  val filterRuntimeData: StateFlow<Boolean> = _filterRuntimeData.asStateFlow()

  设计模式：私有可变 + 公开只读

  ViewModel 内部：
  _filterRuntimeData (MutableStateFlow) ← 可以修改
      ↓
  ViewModel 外部（UI）：
  filterRuntimeData (StateFlow) ← 只能读取，不能修改

  为什么这样设计？
  - ✅ UI 不能直接修改状态（防止混乱）
  - ✅ 只能通过 toggleFilter() 修改（统一入口）
  - ✅ 符合单向数据流原则

  ---
  3. 响应式日志列表

  val logs: StateFlow<List<CommunicationLogEntity>> = _filterRuntimeData
      .flatMapLatest { filter ->
          if (filter) {
              dao.getFilteredLogs()
          } else {
              dao.getAllLogs()
          }
      }
      .stateIn(
          scope = viewModelScope,
          started = SharingStarted.WhileSubscribed(5000),
          initialValue = emptyList()
      )

  工作流程：

  1. _filterRuntimeData 变化（true/false）
     ↓
  2. flatMapLatest 切换数据源
     ├─ false → dao.getAllLogs()（所有日志）
     └─ true → dao.getFilteredLogs()（过滤后的日志）
     ↓
  3. stateIn 转换为 StateFlow
     ↓
  4. UI 自动更新

  类比：
  就像电视遥控器：
  - 按频道 1 → 显示频道 1 的内容
  - 按频道 2 → 自动切换到频道 2 的内容
  - 不需要手动刷新屏幕

  flatMapLatest 解释

  _filterRuntimeData.flatMapLatest { filter ->
      if (filter) {
          dao.getFilteredLogs()  // 返回 Flow<List<...>>
      } else {
          dao.getAllLogs()       // 返回 Flow<List<...>>
      }
  }

  作用：根据条件切换 Flow 数据源

  "Latest" 的含义：
  - 如果快速切换过滤状态
  - 只保留最新的 Flow
  - 取消之前的 Flow

  stateIn 解释

  .stateIn(
      scope = viewModelScope,
      started = SharingStarted.WhileSubscribed(5000),
      initialValue = emptyList()
  )

  参数说明：

  ┌──────────────┬──────────────────────────────────────────┐
  │     参数     │                   说明                   │
  ├──────────────┼──────────────────────────────────────────┤
  │ scope        │ 生命周期范围（ViewModel 销毁时自动取消） │
  ├──────────────┼──────────────────────────────────────────┤
  │ started      │ 何时开始收集数据                         │
  ├──────────────┼──────────────────────────────────────────┤
  │ initialValue │ 初始值（空列表）                         │
  └──────────────┴──────────────────────────────────────────┘

  SharingStarted.WhileSubscribed(5000)：
  - 有订阅者时开始收集数据
  - 最后一个订阅者取消后，等待 5 秒
  - 5 秒内如果有新订阅者，不重新开始（节省资源）
  - 5 秒后仍无订阅者，停止收集
```

## suspend

`suspend` 是 Kotlin 中用于声明**挂起异步函数**的关键字。

使用 `suspend` 修饰函数，意味着该函数可以在**协程（Coroutine）** 中运行，并且不会阻塞主线程（UI 线程），从而避免界面卡顿。

### 1. 只能在协程中调用

`suspend` 函数不能直接在普通函数中调用，只能在协程或者其他 `suspend` 函数中调用。

❌ 错误示例

```kotlin
fun someFunction() {
    dao.insert(log) // 编译错误！
}
```

因为 `insert()` 是 `suspend` 函数。

------

✅ 正确示例

```kotlin
fun someFunction() {
    viewModelScope.launch { // 启动协程
        dao.insert(log)
    }
}
```

通过 `launch` 启动协程后，就可以调用 `suspend` 函数。

------

### 2. 可以“暂停”和“恢复”

`suspend` 函数在执行过程中可以暂停，等待异步任务完成后再继续执行。

### 示例

```kotlin
suspend fun fetchData() {
    val data1 = api.getData1() // 暂停，等待网络请求

    // 收到响应后，自动恢复执行

    val data2 = api.getData2() // 再次暂停

    // 收到响应后，继续执行

    return combine(data1, data2)
}
```

执行流程：

1. 调用 `getData1()`
2. 协程暂停
3. 网络请求完成后恢复
4. 调用 `getData2()`
5. 再次暂停
6. 请求完成后继续执行

整个过程是异步的，但代码写法看起来像同步代码。

------

### 3. 不阻塞线程

`suspend` 的“暂停”并不会阻塞线程。

### 示例

```kotlin
suspend fun example() {
    delay(1000) // 暂停 1 秒，但不阻塞线程
}
```

这里的 `delay()`：

- 会暂停当前协程
- 不会阻塞线程
- 线程在这 1 秒内可以执行其他协程任务

因此协程比传统线程更加轻量。

------

### 总结

`suspend` 的核心作用：

- 支持协程异步执行
- 避免阻塞 UI 线程
- 可以暂停和恢复执行
- 代码写法像同步，但底层是异步

因此在 Android 开发中：

- 数据库操作
- 网络请求
- 文件 IO

通常都会使用 `suspend` + Coroutine 来实现异步处理。

## Result<T>

是一个**密封联合类型（Sealed-like structure）**，它只会有两种状态：

1. **Success(value)**: 包含操作成功的返回值。
2. **Failure(exception)**: 包含操作失败时的异常对象。



创建方式：

```
val success = Result.success("你好")
val failure = Result.failure<String>(Exception("出错了"))
```

一般是网络请求或者runCatching来创建



## runCatching

```
val result: Result<String> = runCatching {
    // 可能会抛出异常的代码
    "Success Data" 
}
```

它本质上是对 `try-catch` 的一层超薄包装，但它改变了代码的**组织结构**：将“捕获异常”这件事从一种语法结构变成了一个**可传递的对象**

像是一个**安全包装盒**。

- **逻辑**：它执行代码块。
- **成功**：把结果装进 `Result.Success`。
- **失败**：把捕获到的异常（Throwable）装进 `Result.Failure`。

### 链式调用

```
runCatching { JSON.parse<User>(json) }
    .mapCatching { user -> db.save(user) } // 如果 save 出错，也会进入 onFailure
    .onSuccess { println("全流程成功") }
    .onFailure { e -> println("任一环节失败: ${e.message}") }
```

### 兜底

两个兜底方案，但用法场景不同：

| **方法**                             | **参数类型**           | **特点**                                                     |
| ------------------------------------ | ---------------------- | ------------------------------------------------------------ |
| **`getOrDefault(value)`**            | 一个具体的**静态值**   | 简单直接，不关心失败原因，只想要个默认数据。                 |
| **`getOrElse { throwable -> ... }`** | 一个 **Lambda 代码块** | **动态兜底**。你可以根据不同的错误类型（throwable）返回不同的默认值。 |

```
// 场景：无论什么错，都给 0
result.getOrDefault(0)

// 场景：如果是空指针给 0，如果是格式错误给 -1
result.getOrElse { e ->
    if (e is NullPointerException) 0 else -1
}
```

**避坑指南**

在使用 `getOrDefault` 时，有一点需要注意：**默认值是立即计算的。**

如果你提供的默认值需要经过复杂的计算或者耗时的操作，建议使用 `getOrElse`。因为 `getOrDefault` 里的默认值即便在成功的情况下也会被先计算出来（虽然不一定用到），而 `getOrElse` 的 Lambda 只有在**真的失败时**才会执行。

Kotlin

```
// 不推荐：即使成功了，也会执行这段耗时创建
result.getOrDefault(VeryExpensiveObject()) 

// 推荐：只有失败时才执行
result.getOrElse { VeryExpensiveObject() } 
```

**总结：** `getOrDefault` 是最简单的“降级处理”。当你不在乎失败的具体原因，只想要程序能拿着一个安全值继续跑下去时，它是你的首选。

### 使用时的注意事项

在 Kotlin 协程中，取消协程是通过抛出 `CancellationException` 实现的。 由于 `runCatching` 会捕获**所有** `Throwable`，它会不小心把协程的“取消信号”也给吞了。

- **后果**：该被取消的协程会继续运行，导致内存泄漏或逻辑错误。
- **对策**：在 `onFailure` 中重新抛出取消异常，或者自定义一个 `runCatchingSafe`。

Kotlin

```
// 正确做法
runCatching { someCoroutineTask() }
    .onFailure { if (it is CancellationException) throw it } 
```

**❌ 坑 2：性能开销**

`runCatching` 是一个 inline 函数，但它涉及 `Result` 对象的创建。在**极端高性能**（如每秒循环千万次）的场景下，原始的 `try-catch` 或简单的 `if` 判断性能更好。当然，对于 99% 的应用开发（网络、UI、数据库），这个开销可以忽略不计。

### 使用场景

**推荐使用**：

- 调用第三方不稳定的 SDK/库。
- JSON 解析、文件读取、网络请求。
- 需要对多个错误步骤进行链式补救时。

**不推荐使用**：

- 协程内部（除非你手动处理了 `CancellationException`）。
- 业务逻辑中的预期内控制流（比如 `if (user == null)` 建议直接判断，而不是抛异常再捕获）。

## 阴影效果

### 1. 使用 `Modifier.shadow` (最通用的方法)

这是最直接的方式。你可以给任何 Modifier 链加上 `.shadow()`。

Kotlin

```
Box(
    modifier = Modifier
        .size(100.dp)
        .shadow(
            elevation = 8.dp,           // 阴影的大小/深度
            shape = RoundedCornerShape(12.dp), // 阴影必须配合形状，否则会切断或显示异常
            clip = false                // 是否裁剪超出阴影的部分
        )
        .background(Color.White, RoundedCornerShape(12.dp))
)
```

------

### 2. 使用 `Surface` 组件

如果你觉得 `Card` 太重（带有很多默认样式），但又想要那种 Material 风格的阴影和处理，可以使用 `Surface`。`Card` 本质上就是对 `Surface` 的一层薄封装。

Kotlin

```
Surface(
    modifier = Modifier.padding(8.dp),
    shape = CircleShape,
    shadowElevation = 4.dp, // 设置阴影
    tonalElevation = 2.dp,  // M3 特色：色调海拔（根据高度改变背景色调）
    color = Color.White
) {
    // 内部内容
}
```

------

### 3. 特殊方式：`Modifier.graphicsLayer`

如果你需要更精细地控制阴影颜色（例如彩色阴影），可以使用 `graphicsLayer`。

Kotlin

```
Box(
    modifier = Modifier
        .size(100.dp)
        .graphicsLayer {
            shadowElevation = 20f.dp.toPx() // 阴影高度
            shape = CircleShape            // 阴影形状
            clip = true
            spotShadowColor = Color.Red    // 关键点：可以自定义阴影颜色！
            ambientShadowColor = Color.Red
        }
        .background(Color.White, CircleShape)
)
```

------

### 核心区别：为什么推荐在 Card 中使用？

虽然哪里都能加，但在 `Card` 中加阴影有几个天然优势：

1. **自动适配状态**：`CardDefaults.cardElevation` 允许你一次性定义“默认、按下、聚焦”等多种状态下的阴影，它会自动帮你做动画过渡。如果你用 `Modifier.shadow`，这些交互动画需要你自己写 `animateDpAsState`。
2. **符合规范**：`Card` 的阴影默认遵循 Material Design 3 的光影物理模型，看起来更自然。
3. **性能优化**：`Card` 内部对层级渲染做了优化，避免了在复杂列表里因大量阴影导致的掉帧问题。



### 按钮阴影

```
modifier = Modifier.shadow(
    elevation = Dimension.Elevation.medium,
    shape = MaterialTheme.shapes.medium
)
```

或者

```
elevation = ButtonDefaults.buttonElevation(
    defaultElevation = Dimension.Elevation.small,
    pressedElevation = Dimension.Elevation.medium
)
```



### 总结

- **想简单快速、有交互动画**：用 `Card` 或 `ElevatedCard`。
- **想在非卡片组件上加（如图片、圆形头像）**：用 `Modifier.shadow`。
- **想做酷炫的彩色阴影**：用 `Modifier.graphicsLayer`

# 踩坑

#### 高度被恶意撑开

父节点布局是

```
wrapContentHeight()
```

子内容的布局用了

```
fillMaxSize()
```

然后就无限撑开了；

正常的配置应该是

```
wrapContentHeight() / fillMaxWidth()
```

```
weight(1f) / fillMaxSize()
```
